# Constrained answer design for the Almere D6 open-question set

## Executive conclusion

The repository’s current review state is already narrow enough to move from “open-ended stakeholder interviews” to structured validation instruments. The active review entry points now are the current learnings note and the current open-questions note, and the repository explicitly says that no D6 row may move to `settled` from those notes alone. The current position is that there are 12 D6 responsibility rows, 11 already have public-source prefill, none are `settled`, and the remaining work is validation rather than broad public search. citeturn9view0turn10view0turn7view4

The practical implication is strong: almost every unresolved row can be converted into one of a small number of answer formats — an assurance question, a role matrix, a component inventory table, a funding matrix, or a short decision tree. What still cannot be settled from the repository alone are the same dimensions the current notes keep flagging: formal D6 classification, owner/coordinator distinction, mandate, component-level funding, continuity funding, ICT/privacy accountability, and safe bestuurlijke wording. citeturn6view0turn7view4turn11view0

The best design choice is therefore not “ask stakeholders to explain everything.” It is “ask narrow, prefilled, evidence-attached questions.” The main exceptions are the broad cross-cutting rows — especially digital/operational infrastructure, the PGA/Zorgzaam/Flever interface, inloop/social-base rows, and financing — which should be split or matrixed before anyone tries to validate them. The repository itself already points in that direction by saying that digital/operational infrastructure is not one public object, that the PGA/Zorgzaam/Flever interface needs actor separation, and that the funding problem is a matrix problem rather than a generic public-search problem. citeturn10view1turn11view0turn6view0

## Controlled vocabularies and evidence gates

The current active notes already define the recurring unresolved dimensions: classification, owner/coordinator/executor, mandate, funding line, continuity, scale, ICT/privacy accountability, and safe wording. They also say the register may move only when evidence directly supports classification, owner, executor, scale and funding, and that disagreement should be kept as `review_needed`. That makes controlled vocabularies and evidence gates the right next step. citeturn6view0turn7view4turn11view0

**Evidence gates**

1. **Scope gate:** if a row covers multiple facilities, actors, subcomponents or scales, split it before asking for validation. This is especially necessary for digital/operational infrastructure, PGA/Zorgzaam/Flever, inloop/social-base, informal support and financing. citeturn10view1turn11view0
2. **Authority gate:** if there is a direct local or other authoritative source that answers the question, treat it as **A**. If there is only public-source-backed prefill, treat it as **B**. If the answer depends on a local decision, controller matrix, mandate record or privacy arrangement, treat it as **C**. citeturn7view4turn6view0
3. **Assurance gate:** when the repository already has a plausible inference, the stakeholder form should ask “Is the repository inference correct?” rather than “Please explain.” citeturn10view0turn10view1
4. **Settlement gate:** no row should move to stronger wording or `settled` unless the evidence directly supports classification, owner, executor, scale and funding. citeturn11view0
5. **Conflict gate:** if answers conflict, the repository rule should be `review_needed`, not forced synthesis. citeturn11view0
6. **Funding gate:** if component-level funding or continuity cannot be evidenced, keep funding unresolved and escalate to finance/controller. citeturn6view0turn10view1

## Register-update rules

The current repository guidance is explicit: the notes are for validation preparation, not settlement; stakeholder answers should not be filled by inference; and any move to stronger wording requires direct evidence. These rules can therefore be operationalised into a small register-update logic. citeturn9view0turn11view0

1. **Use A/B/C at question level, not only row level.**  
   - **A:** direct authoritative answer exists; prefill may be strong.  
   - **B:** repository inference exists; use assurance question.  
   - **C:** local decision, finance, mandate or ICT/privacy validation is required.

2. **Do not ask free-text questions when a finite option set exists.**  
   Most current rows can be validated with one of five instruments: assurance question, role matrix, component inventory table, funding matrix, or decision tree. citeturn10view0turn10view1

3. **Split broad rows before updating the register.**  
   This is especially required where the repository already says the row is not one public object or contains multiple actor/scale layers. citeturn10view1turn11view0

4. **Only mark `settled` when all core fields are evidenced together.**  
   Minimum evidence bundle: classification, owner/coordinator distinction, mandate/decision basis, scale, and funding. If one fails, keep `inferred`, `unknown`, or `review_needed`. citeturn11view0

5. **Use answer types to drive register effect.**  
   - `Formal D6` + full evidence bundle → potentially `settled`.  
   - `Adjacent` / `Implementation context` → keep linked but not settled as D6.  
   - `Candidate upgrade` → keep inferred or create decision-needed item.  
   - `Mixed/split required` → split row before any stronger claim.  
   - `Unknown` → no change.

6. **Treat funding as a separate gate.**  
   No continuity claim, no structural-funding claim, and no component allocation claim should be written without controller-backed evidence. citeturn6view0turn10view1

7. **Treat digital/accountability as a separate gate.**  
   If the row has data/monitoring/operational content and data/privacy accountability is not attached, escalate to ICT/privacy owner. citeturn6view0

8. **Handle conflicting answers explicitly.**  
   Record competing answers, keep the evidence field, mark `review_needed`, and escalate to the appropriate owner rather than reconciling by inference. citeturn11view0

### Remaining questions that cannot be constrained much further

Some questions can be narrowed, but not answered, until internal evidence exists. The current open-questions note and boundary note are explicit about this. citeturn6view0turn7view4

- **Formal D6 classification** remains a local decision or validation-record question across nearly all rows. The answer space can be constrained, but the underlying decision may still not exist publicly. citeturn6view0turn11view0
- **Owner/coordinator distinction** can be reduced to a role matrix, but not filled safely from public actor visibility alone. citeturn6view0turn10view1
- **Mandate and decision route** often remain irreducibly internal where public documents show a project or programme but not the D6 mandate. citeturn6view0turn7view4
- **Component-level funding and continuity funding** remain finance/controller tasks even if the answer options are tightly constrained in a matrix. citeturn6view0turn10view1
- **ICT/data/privacy accountability** for digital, monitoring and dashboard rows cannot be inferred safely without internal governance evidence. citeturn6view0
- **Safe final bestuurlijke wording** is still an approval question, not a source-extraction question. citeturn6view0turn7view4

The net result is useful: the open-question set is no longer too broad. It can be turned into rapid, evidence-attached stakeholder validation forms. What still remains is not a lack of answer space, but a lack of validated answers. citeturn10view0turn11view0
